# CashBack Pro

Семейное приложение для управления кешбеком и банковскими картами.
React + TypeScript + Vite + Supabase, дизайн в стиле iOS 26 "Liquid Glass".

## Стек

- React 19 + TypeScript + Vite
- Supabase (Auth + Postgres + RLS)
- TanStack Query
- Tailwind CSS v4
- lucide-react
- tesseract.js (OCR в браузере)

## Исправление RLS в Supabase

Если возникает ошибка `infinite recursion detected in policy for relation "profiles"` или `new row violates row-level security policy for table "families"` — откройте **SQL Editor** в Supabase и выполните файл:

```
supabase/fix_rls_recursion.sql
```

**Причина бага:** политика `"Users can view family member profiles"` на таблице `profiles` искала `family_id` пользователя через подзапрос `select family_id from profiles where id = auth.uid()` — то есть запрос к `profiles` внутри политики, определённой на самой `profiles`. Postgres обязан применить RLS и к этому внутреннему запросу, что и порождает бесконечную рекурсию. Почти все остальные политики (`families`, `family_members`, `cards`, `cashback_categories`) резолвили "мою семью" через такой же подзапрос — поэтому рекурсия "протекала" и туда, в том числе мешая вернуть только что вставленную строку `families` (отсюда и ошибка при создании семьи).

**Решение:** вынесли резолв `family_id` в функцию `public.current_family_id()` с `security definer` — она выполняется с правами владельца функции и не проходит через RLS таблицы `profiles`, разрывая рекурсию. Скрипт идемпотентен — можно выполнять повторно.

`supabase/schema.sql` обновлён той же логикой — на новых проектах баг не появится изначально.

## Распознавание категорий со скриншота (OCR)

В исходном техзадании предполагалась библиотека **sharp** для предобработки изображения. Sharp — это Node.js-библиотека с нативными бинарными зависимостями, она физически не может работать в браузере, а это приложение — чистый SPA без своего сервера. Поэтому тот же пайплайн (увеличение/изменение размера → ч/б → контраст → резкость) реализован через Canvas API прямо на клиенте (`src/shared/lib/ocr.ts`), а распознавание текста выполняет **tesseract.js** (WebAssembly-сборка Tesseract OCR, языки rus+eng) — тоже полностью в браузере, без сервера.

Раньше загрузка скриншота была полностью замокана (`setTimeout` + три фиксированные категории независимо от картинки) — поэтому распознавание "не работало". Теперь это настоящий OCR: текст разбивается на строки, в каждой ищется процент (`\d+%`), а оставшаяся часть строки становится названием категории; строки без `%` (заголовки, подсказки вида "для зарплатных клиентов") отфильтровываются автоматически.

Иконки категорий, чекбоксы и значки ⓘ/®/© на скриншоте банка OCR читает как мусорные символы или одиночные буквы, прилипшие к началу/концу строки (например `"О Деливери Фе О"` вместо `"Деливери"`). Парсер разбивает строку на слова и итеративно отрезает с обоих краёв токены, которые — почти наверняка не часть названия: либо это символ без единой буквы/цифры (`@`, `®`, `©`, `№`, `+)`...), либо слово из 1-2 букв (`О`, `Фе`, `ФФ`, `8`...). Резать перестаёт, как только остаётся «нормальное» слово — поэтому реальные короткие названия (например `АЗС`) не страдают.

OCR-модуль подключается отдельным чанком (`import()` по требованию) — он не увеличивает размер основного бандла, пока пользователь не нажмёт «Выбрать скриншоты».

## Деплой на GitHub Pages

Если настроить Pages на «Deploy from a branch», GitHub просто отдаёт файлы репозитория как есть — без сборки. Для Vite-приложения это означает, что браузер получает `index.html`, который ссылается на несобранный `src/main.tsx`, а такой файл браузер не может выполнить — отсюда пустой белый экран (заголовок вкладки рендерится, потому что это обычный HTML, а весь JS — нет).

В проекте уже есть workflow `.github/workflows/deploy.yml`, который собирает проект (`npm run build`) и публикует именно `dist/`. Нужно один раз переключить настройку:

1. В репозитории: **Settings → Pages → Build and deployment → Source** → выбрать **GitHub Actions** (вместо «Deploy from a branch»).
2. Запушить любой коммит в `main` (или запустить workflow вручную через **Actions → Deploy to GitHub Pages → Run workflow**).
3. Сайт соберётся и опубликуется автоматически на `https://badik2001.github.io/cashback-pro/`.

Что было поправлено для этого:
- `vite.config.ts` — `base: '/cashback-pro/'`, иначе собранные `<script>`/`<link>` ссылались на корень домена и 404-лись.
- `BrowserRouter` в `App.tsx` теперь получает `basename`, синхронизированный с этим же base — внутренняя навигация работает корректно из под подпути.
- В workflow добавлено `cp dist/index.html dist/404.html` — у GitHub Pages нет серверного роутинга, без этого обновление страницы на `/cashback-pro/cards` дало бы 404.
- `.env.production` закоммичен в репозиторий (это публичный anon-ключ Supabase, не секрет — он и так уходит в браузер любого пользователя, защищён через RLS) — Action подхватывает его при сборке без дополнительной настройки GitHub Secrets.

Если переименуете репозиторий или захотите деплоить на кастомный домен/в корень — пересоберите с `VITE_BASE_PATH=/ npm run build` (или с нужным путём) либо поменяйте значение в `vite.config.ts`.

**Supabase Auth:** если будете отправлять письма с подтверждением/восстановлением, добавьте `https://badik2001.github.io/cashback-pro/` в Supabase → Authentication → URL Configuration → **Redirect URLs** (иначе Supabase отклонит редирект на незнакомый домен).
